# PlayMCP 등록 및 예선 제출 체크리스트

> 새 기능 추가 없음. 등록·검증·제출 준비 절차만 정리한다.

## 0. 현재 구현 상태 요약

- MCP tool 6개 구현 완료, vitest 15개 통과, `tsc --noEmit` 에러 없음, 실제 MCP stdio
  클라이언트로 데모 플로우 1회 완주 확인됨.
- **DB는 "구현 완료"가 아니라 "migration 준비 완료" 상태다.**
  `migrations/0001_chaeggimi_mvp.sql`은 작성되었으나 실제 Supabase/Postgres에 적용된
  적이 없고, `src/store.ts`는 여전히 **인메모리 store**다.

## 1. 영속성 결정 (확정, 2026-06-28)

```text
예선 MVP는 인메모리 store 기준으로 동작한다.
서버 재시작 시 데이터는 유지되지 않는다.
이는 3분 데모와 예선 심사용 범위에서는 허용 가능한 제한으로 본다.
본선 진출 시 Supabase/Postgres 연결을 통해 데이터 영속성을 확보한다.
```

- "DB migration 작성 완료"(`migrations/0001_chaeggimi_mvp.sql`)와 **"Supabase 연결
  완료"는 서로 다른 상태**다. 현재는 전자만 완료되었고, 후자는 본선 진출 확정 후
  착수하는 후속 작업으로 분리한다.
- PlayMCP가 등록 시 24시간 상시 외부 엔드포인트 구동을 요구하는지는 [미확인]이다.
  만약 그렇다면 재시작 빈도가 늘어 인메모리 리스크가 커지므로, **3번 등록 절차를
  실제로 진행하며 반드시 이 부분을 확인**한다.

## 2. 인메모리 store 리스크 상세 (등록 전 반드시 인지)

| 항목 | 현재 상태 | 영향 |
|---|---|---|
| 데이터 저장 위치 | Node 프로세스 메모리 (`ChaeggimiStore` 클래스 내부 배열) | 서버 프로세스가 재시작되면 등록된 예약/약/공유설정이 **전부 소실** |
| 다중 사용자 | `DEMO_USER_ID` 고정값 사용 | 실제 여러 사용자가 동시에 쓰면 모든 사용자가 같은 데이터를 공유함 — **실사용 불가, 데모 전용** |
| PlayMCP 콘솔 재배포 | 코드 배포/재시작마다 데이터 초기화 | 데모 시연 직전에 한 번 정상 흐름을 미리 확인해두는 것을 권장 |

**결론**: 지금 상태로는 "예선 데모 시연"까지는 문제없지만, **PlayMCP에 전체공개로
등록해서 다른 사용자(심사위원/투표자)가 실사용하게 하려면 영속 저장소가 필요**하다.
예선 접수 시점에 데모만 시연하는 것이라면 인메모리로 충분할 수 있으나, 본선 진출 시
(투표 기간 8/31~9/28) 실사용자가 직접 써보는 구조라면 그 전까지는 Supabase 연결이
필수다.

## 3. PlayMCP 등록 절차 (등록 단계)

1. 카카오 개발자 계정으로 PlayMCP(playmcp.kakao.com) 로그인
2. MCP 서버 배포 위치 준비 — 현재는 로컬 stdio 서버(`node dist/index.js`)이므로,
   PlayMCP에 등록하려면 **카카오클라우드 등 외부에서 접근 가능한 엔드포인트로
   배포**해야 함 (stdio → HTTP/SSE 등 PlayMCP가 요구하는 연결 방식 확인 필요,
   [미확인] — 콘솔에서 직접 확인)
3. PlayMCP 콘솔에서 서버 등록 (엔드포인트 URL 입력)
4. tool 6개가 정상적으로 `tools/list`에 노출되는지 콘솔에서 확인
5. 데모 플로우(`docs/demo-cases.md`) 1회 직접 실행해서 정상 동작 확인
6. 상태를 **"나에게만 공개" → "전체 공개"로 전환** (신청 자격 필수 조건)
